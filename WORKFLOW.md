# CCFA-Skill 完整流程详解

> 本文档详细介绍 CCFA-Skill 六个阶段中，Claude 在每一步会做什么、如何与你交互。
> 概览和安装见 [README](README.md)。

整个流程通过对话自然推进，无需记忆切换命令。Claude 在每个关键节点主动询问你是否继续，未经你确认绝不跳到下一阶段。

六个阶段及其产物：

| 阶段 | 名称 | 主要产物 |
|------|------|---------|
| A | 方向探索与调研 | `docs/idea_report.md` Part 1 |
| B | Idea 深化 | `idea_report.md` Part 2 |
| C | 实验设计 | `idea_report.md` Part 3 |
| D | 实现设计 | `docs/implementation.md` |
| E | 编码 | 代码 + `docs/dev_log.md` |
| F | 论文撰写 | `docs/manuscripts/` 下的论文（按版本存档）|

---

## 贯穿阶段 A / B 的「确认卡片」

在阶段 A 和阶段 B，Claude 的**每一次输出都以一张「已确认内容卡片」开头**，让你随时看到当前已锁定的共识：

```
━━━━━━━━━━ 已确认内容 ━━━━━━━━━━
研究方向：……
主 RQ：……
次 RQ：……
方向约束：……
RQ 约束：……
参考论文：……
技术框架：……（阶段 B）
Pipeline：……（阶段 B）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

卡片只列**已被你确认且非空**的字段——未确认或无内容的字段整行省略，不写「待确认」「无」之类占位；流程最开始没有任何已确认内容时，不显示卡片。候选方向、过程中的检索清单都不进卡片；只有你明确点名要参考的论文才会列入。卡片内容与 `docs/user_requirements.md` 实时同步。

## 阶段 A：方向探索与调研

> 目标：通过充分的多轮交互，把一个模糊的研究兴趣，收敛成一个客观成立的研究方向和一组明确的研究问题（RQ）。

### Claude 会做什么

**1. 解析输入、收集需求**
从你的输入中提取研究方向、已有想法、参考论文和约束。信息不足时一次性提问（不分多轮），收集后写入 `user_requirements.md`，区分「方向约束」和「RQ 约束」。

**2. 初步文献检索**
优先检索顶会/顶刊（NeurIPS、ICML、ICLR、CVPR、ACL 等），目标不少于 15 篇有效文献。每个 research gap 要求 ≥2 篇论文支撑，不足则补充检索（最多 3 轮）。检索不足时会如实告知原因，并问你是否继续。

**3. 确认下载清单 → 下载论文**
列出建议下载的论文清单（标题、发表信息、arXiv 版本、一句话说明、下载原因），你确认后批量下载到 `docs/papers/`。优先 arXiv，找不到自动尝试 OpenReview，都失败则保存摘要 TXT。

**4. 锚定问题域，逐个确认研究方向**
不会一次性抛 5 个方向让你选，而是：
- 先从三个维度汇报问题域全貌（这个方向主要解决什么、主流方法分几类、近两年最活跃的子方向）
- 根据你的兴趣，一次聚焦 1–2 个候选方向深入讨论
- 来回交互，直到你明确选定一个方向并锁定

**5. 逐个提炼并确认 RQ**
- 一次提出一个主 RQ 候选，附：对应哪个 gap、新颖性核查（专项搜索结果）、可回答性评估
- 与你讨论，可改写/合并/替换，直到主 RQ 确认
- 再按同样方式逐个确认次 RQ（1–3 个）
- 全部确认后，撰写必要性论证（应用/理论/时机三点，每点附论文支撑）

**6. 汇编 Part 1**
方向、RQ、必要性论证全部确认后，征得你同意，把这些内容汇编成 `idea_report.md` Part 1（Motivation、Research Questions、Key Works），进入阶段 B。

### 你需要做什么

- 回答 Claude 的澄清问题，表达方向偏好和约束
- 在每个候选方向/RQ 上给出选择或修改意见
- 挑战 Claude 给出的 gap 判断——每个 gap 都有论文证据，你评判的是「证据够不够」
- 确认方向和 RQ 已锁定

## 阶段 B：Idea 深化

> 目标：方向和 RQ 已明确，把「研究问题」深化为「可实现的方法」。同样每次输出都带确认卡片，分三层逐步确认。

### Claude 会做什么

**0. 汇编并审阅 Part 1**
把阶段 A 的成果汇编为 Part 1，展示供你审阅，确认无误再继续。

**1. 确认技术框架（第一层）**
先和你对齐「用什么大方向的技术来回答 RQ」，给出整体思路、框架草图（输入→模块→输出）、每个模块解决 RQ 的哪一部分。不涉及实现细节，来回交互直到框架确认。

**2. 确认详细 pipeline（第二层，大白话）**
把框架细化为可执行的完整流程。**输出是大白话，用「第一步…第二步…」讲清每个环节在做什么，不堆砌公式。** 每步还会说明设计的直觉理由。确认后才落笔 Part 2 的 Method 章节（3.2 大白话流程详解、3.3+ 对应理论与公式）。

**3. 引用内容核验**
对每条来源批注，打开对应 PDF 定位支撑段落，摘录原文附在批注后。无法核验的标注 `⚠️ [低置信度：PDF 不可用]` 并登记待核实清单——不确定性显式传递，绝不掩盖。

**4. Introduction 精修（第三层）**
最后专门打磨 Introduction，按论文风格撰写（领域重要性→现有方法局限→本文动机→方法概述→分点贡献），达到可投稿质量，与你确认逻辑和力度。

**5. 进入阶段 C**
技术框架、pipeline、Method、Introduction 全部确认后，征得同意进入实验设计。

### 你需要做什么

- 在技术框架、pipeline、Introduction 三个节点分别确认或提出调整
- pipeline 是大白话描述，重点检查流程有没有缺漏或不合理
- 确认 idea 已深化完整

## 阶段 C：实验设计

> 目标：先精读 baseline 的真实实验设计，再据此设计出符合领域惯例、可执行的完整实验方案。
> **核心原则**：实验设计的第一目的是严格证明 idea 的有效性，不为迁就资源而削减实验。因此不在设计前收集 GPU/训练时间约束，而是在方案完成后给出资源预估供参考。

### Claude 会做什么

**1. Baseline 论文与代码精读**
这是实验设计的关键前置环节：
- 先向你展示精读计划：打算读哪些 baseline 的论文和 GitHub、它们的核心 idea 是什么、为什么要读
- 你确认后，逐一精读论文 PDF 和代码仓库
- 从中提取每个 baseline：用了哪些数据集、怎么划分、设计了哪些实验、对比了哪些模型、用了哪些指标、关键超参数
- 整理为 `idea_report.md` Part 3 第 0 节

**2. 领域惯例归纳**
综合精读结果，归纳各 baseline 共同使用的标准 benchmark、标准指标、消融设计惯例和结果汇报规范，作为后续实验设计的参考基准。

**3. 数据与代码可获取性核实**
只核实「实验能否做得成」——确认数据集链接可访问、baseline 代码仓库可访问。任一项不通过则暂停并告知你。**不核实 GPU/显存是否够用**。

**4. 提出实验大纲，与你确认**
动笔写正文前，先给出一份实验大纲与你交互：用什么数据集、几个实验各自如何设计、为什么这样设计、每个实验有多少模型参与、核心 baseline 是什么及选它的理由。同时列出多个可选的扩展实验供你勾选。大纲确认后才展开写完整 Part 3。

**5. 生成实验设计**
按确认的大纲、顶会标准工作量展开：主实验 3–5 个数据集 + 5–8 个 baseline；消融实验系统覆盖所有创新模块（3–6 个变体）；附加实验分两类——领域多篇论文重复出现的标配实验（必做）+ 你选定的扩展实验。每个实验都说明「为什么这样设计、对支撑核心方法有什么意义」；参与评测的模型逐个说明与本文方法的区别、纳入意义、出处论文（仅展示），并指明核心 baseline。所有实验报告均值±标准差，至少 3 次随机种子。

**6. 给出资源预估（仅供参考）**
方案设计完成后，基于模型规模和实验组数给出一份资源预估（预估显存、单次训练时长、实验组数），写入 Part 3 末尾。**这只是参考，不是设计约束**；资源受限时可优先跑核心实验、分批运行。

### 你需要做什么

- 确认或调整 baseline 精读清单
- 审查实验大纲，从可选扩展实验中勾选要做的
- 审查完整实验设计，提出修改意见

## 阶段 D：实现设计

> 目标：生成精确到每个函数的编码指南 `implementation.md`，作为阶段 E 的施工图。
> `implementation.md` 开头有两个关键部分：**完整目录树** + **逐文件功能表**。

### Claude 会做什么

**1. 选择实现策略**
- **强 Baseline 路径**（基于已有开源项目）：git clone 原始项目 → 扫描现有结构 → 生成改写方案，每个需修改的函数说明原来做什么、改为做什么、参数/返回值变化
- **从头构建路径**：完整目录树 + 逐文件功能表；独立数据流章节（原始文件→解析→划分→标准化→模型输入 tensor，含 shape）；每个函数说明签名、参数、返回值、实现逻辑

**2. 自动校验**
每次生成或修改 `implementation.md` 后，自动执行三项校验：实验覆盖（Part 3 每个实验都有对应实现入口）、逻辑一致性（tensor shape 跨模块对齐）、完整性（每个文件都有说明）。

**3. 编码前确认清单**
`implementation.md` 确认后，开始编码前逐项与你确认：① 用什么环境、环境名是什么（能查到则复用，查不到则新建）；② 设备对环境的特殊要求；③ 数据集是否已下载——下载快的 Claude 自己下，慢的给你命令和路径你自己下；④ 代码要不要 Claude 自动运行（按预计运行时间建议：快的 Claude 跑、慢的你跑、混合）；⑤ README 放项目根目录还是 `code/`。

### 你需要做什么

- 说明是从头构建还是基于已有开源项目（提供链接）
- 审查实现方案
- 回应编码前确认清单（环境、设备、数据集、运行策略、README 位置）

---

## 阶段 E：编码

> 目标：按 `implementation.md` 逐文件实现，全程同步维护开发日志、README 和可视化 notebook。

### Claude 会做什么

**1. 先建 README 和 notebooks/**
编码前先建立两样贯穿全程的产物：**README.md**（项目内容 + 环境配置 + 详细运行命令，放在你确认的位置）和 **notebooks/**（为每个关键步骤建一个 Jupyter 文件做可视化）。

**2. 逐文件实现**
按实现顺序逐个文件编码，每完成一个文件立即更新 `dev_log.md`；影响运行/环境的同步更新 README，关键步骤同步补 notebook。

**3. 按运行策略执行**
按你确认的策略：Claude 自动运行 / 你运行 / 混合（快的 Claude 跑、慢的你跑）。`✅ Done` 只在运行验证通过后标记。

**4. 模块校验与错误处理**
每完成一个模块做一致性校验（函数签名、tensor shape、指标）。若发现 `implementation.md` 有逻辑错误或无法实现，**不会擅自改代码绕过**，而是停止 → 报告 → 等你确认 → 先改 `implementation.md` → 再改代码。

**5. 完工后主动代码审查**
全部编码完成后，Claude 主动审一遍代码并汇报。这是验证论文 idea 的研究代码，**不做工程级严格审查**（不纠结命名、抽象、性能），只守两条硬底线：① 代码能端到端跑起来；② 逻辑完全正确（与方法设计一致、shape 自洽、指标/消融开关正确、无静默 bug）。触碰这两条的问题必须修（经你确认），仅风格/优化类只提示不主动改。

**6. 改进时同步文档**
你对代码提出改进要求、或 Claude 审查后修复时，每次都同步更新 README（若影响运行/结构/功能）和 `dev_log.md`（新增带具体时间的日志条目），三者保持一致。

**7. 依赖规则**
`requirements.txt` 只写库名、不写版本号，且不含 `torch`/`torchvision`/`torchaudio`（由你按 CUDA 版本自行安装）。

### 你需要做什么

- 遇到阻塞性问题时，决定如何解决，或回溯到阶段 C/D
- 按运行策略运行慢速代码，把实验结果反馈给 Claude（可触发结果对比分析）
- 审阅 Claude 的代码审查报告，决定是否修复
- 提出代码改进要求（Claude 会同步更新 README 和 dev_log）

## 阶段 F：论文撰写

> 目标：综合前面所有产物（idea_report、实验结果、文献）撰写论文。论文存放在 `docs/manuscripts/`，**每改一版都复制一个新文件**，版本可追溯。

### Claude 会做什么

**1. 先确认论文结构（第一步，必做）**
写正文前，先和你确认：所有一级/二级标题、每章写什么和写作思路（尤其 Introduction、Related Works 怎么写、为什么这样写）、图表规划（哪里有图/表、各展示什么、怎么设计、哪些手绘哪些 Python 生成）、以及你对论文的额外要求。结构确认后才写初稿。

**2. 生成初稿**
按确认的结构生成 `v1.0-初稿.md`。格式：中文版 skill——标题英文+中文斜体翻译，`Introduction`/`Related Works` 等一级标题英文，其余（二级标题、正文、图注）中文，最多到二级标题；英文版 skill 全英文。每章/节开头用 `>` 标写作思路；每章/节开头、每段结尾、每个图注/表注下都留一个**空白 `>`** 供你批注；图必有图注并注明手绘/Python；最后是 MLA 参考文献，每条下用 `>` 注明贡献和引用原因。

**3. 按批注改稿（循环）**
你在空白 `>` 处写好批注后告诉 Claude「已批注完成」，Claude 自动读取最新版本里你的批注，逐条改稿，**复制为新版本文件**（按改动大小自动定大/小版本号，命名 `v{大}.{小}-{15字简述}.md`），更新标题下的版本批注块（版本号、时间、本版修改说明）。循环直到论文完善。

**4. 引导生成图表**
当论文改得差不多，Claude 主动引导你同意生成 Python 可生成的图/表：图建 `notebooks/image.ipynb`（一个单元格一张图，注明 Fig 几/图注/如何理解，可导出 SVG 到 `notebooks/image/`，导出代码默认注释）；表建 `notebooks/table.ipynb`（一个单元格一张表，格式与论文一致，标注表名和行列含义）。手绘图（如方法框架）由你自己画。

### 你需要做什么

- 确认论文结构（大纲、写作思路、图表规划、额外要求）
- 查阅初稿，在空白 `>` 位置逐处写批注，完成后告诉 Claude
- 同意 Claude 生成 Python 图表；手绘图自己绘制
