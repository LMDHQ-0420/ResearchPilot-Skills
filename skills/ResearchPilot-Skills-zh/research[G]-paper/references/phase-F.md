## 版本与文件命名

`docs/manuscripts/` 下每个论文 md 命名格式：
```
v{大}.{小}-{主要修改内容（15 字以内简述）}.md
```
示例：
- `v1.0-初稿.md`
- `v1.1-补充实验分析.md`
- `v2.0-重构方法章节.md`

**大小版本号由 Claude 根据修改幅度自动判断**：
- **大版本 +1**（v1.x → v2.0）：结构性改动（增删章节、重写整章、方法/实验设计调整）
- **小版本 +1**（v1.1 → v1.2）：局部改动（润色、补段落、改图注、修参考文献）

每次改稿都新建文件，旧版本保留不动。

## 论文格式规范

### 语言（中文版 skill）

- **论文标题**：英文，紧跟一行中文斜体翻译
- **一级标题**：`Introduction`、`Related Works`、`Method`、`Experiments`、`Conclusion` 等保持英文
- **其余全部中文**：二级标题、正文、图注、表注
- 整篇论文**最多只能有二级标题**（`#` 标题 + `##` 一级 + `###` 二级；不得出现 `####`）

> 注：`#` 用于论文标题（全文唯一）；`##` 用于一级章节标题（英文）；`###` 用于二级小节标题（中文）。

### 标题下的版本批注块

论文标题下方，紧跟一个 `>` 批注块，标明本版本信息：
```markdown
# {Paper Title in English}
*{中文标题翻译}*

> **版本**：v1.0
> **时间**：{YYYY-MM-DD HH:MM}
> **本版修改**：初稿
```
第二版起，"本版修改"写清**改了什么、与上一版的区别**，格式统一：
```markdown
> **版本**：v1.2
> **时间**：{YYYY-MM-DD HH:MM}
> **本版修改**：{改了什么，与上一版的区别，一段话；格式与历版一致}
```

### 写作思路批注

- **每一章（一级标题）、每一节（二级标题）的开头**，用 `>` 标明这一章/节的写作思路
- 写作思路批注之后，再正文

### 供用户批注的空白 `>`

为方便用户批注，在以下位置各放一个**空白的 `>`**（仅一个 `>` 符号，后面留空）：
- 每一章开头（写作思路批注之后）
- 每一节开头（写作思路批注之后）
- 每一个自然段的结尾
- 每一个图注 / 表注的下面

> 空白 `>` 是给用户写批注用的占位，Claude 生成初稿时留空，改稿时读取用户在这些位置写入的内容。

### 图

- 每张图**必须有图注**，图注下方跟一个空白 `>`
- 每张图标明是**手绘**还是 **Python 生成**
- 图在正文中按 `Fig. N` 引用

### 表

- 每张表必须有表注（表名 + 每行每列含义）
- 表在正文中按 `Table N` 引用

### 参考文献

- 论文最后是参考文献，正文中引用格式 `[x]`
- 参考文献列表用 `[1]`、`[2]` 编号，**MLA 格式**
- 每条参考文献下方用 `>` 说明两点（各一句话）：
```markdown
[1] {MLA 格式条目}
> 主要贡献：{这篇论文做了什么，一句话}
> 引用原因：{为什么本文引用它，一句话}
```
- 所有文献必须真实存在（经 web_search 验证或来自 `docs/papers/`），不得捏造

## F-1 确认论文结构（第一步，必做）

**在写任何正文之前**，先与用户确认论文结构，展示以下内容并交互：

```
开始写论文前，先确认论文结构：

**整体大纲**（所有一级、二级标题）：
## Introduction
## Related Works
## Method
  ### {二级标题}
## Experiments
  ### {二级标题}
## Conclusion

**各章节写什么、写作思路**：
- Introduction：{主要写什么；写作思路；为什么这样写——如何从领域背景引到 gap 再到本文贡献}
- Related Works：{归纳哪几类相关工作；写作思路；为什么这样组织}
- Method：{...}
- Experiments：{...}
- Conclusion：{...}

**图表规划**：
| 编号 | 类型 | 位置 | 主要展示什么 | 手绘 / Python 生成 |
|------|------|------|------------|------------------|
| Fig. 1 | 图 | Method | {方法框架图} | 手绘 |
| Fig. 2 | 图 | Experiments | {结果曲线} | Python 生成 |
| Table 1 | 表 | Experiments | {主实验对比} | Python 生成 |

- 图：说明每张图展示什么、如何设计；并区分手绘（如方法框架示意）与 Python 生成（如数据曲线、可视化）
- 表：说明每张表展示什么、如何设计

**你对这篇论文还有什么额外要求？**（如目标会议风格、篇幅、强调的卖点等）

确认结构后我开始写初稿。
```

与用户来回交互，直到论文结构（大纲 + 写作思路 + 图表规划 + 额外要求）确认。把额外要求写入 `docs/user_requirements.md` 阶段 F 章节。

## F-2 生成论文初稿

结构确认后，在 `docs/manuscripts/` 生成 `v1.0-初稿.md`：

- 严格按上述**论文格式规范**（语言、标题层级、版本批注块、写作思路批注、空白 `>`、图注、参考文献）
- 按确认的大纲和写作思路逐章撰写
- 图：手绘图在正文留图位 + 图注（注明"手绘"）；Python 生成图先留图位 + 图注（注明"待 Python 生成"），暂不生成图片
- 实验数据来自 `dev_log.md` 已记录的真实结果，无结果的标占位并说明
- 参考文献来自 `docs/papers/` 与正文引用，逐条配 `>` 双行说明

生成后提示用户：
```
论文初稿 v1.0 已生成：docs/manuscripts/v1.0-初稿.md

请查阅初稿，在文中的空白 > 位置写下你的批注修改意见
（每章/节开头、每段结尾、每个图注/表注下方都留了空白 > 供你批注）。
批注完成后告诉我"已批注完成"，我会读取你的批注进行改稿。
```

## F-3 根据批注改稿（迭代循环）

用户告知"已批注完成"后：

1. **自动查阅 `docs/manuscripts/` 下最后一个版本**的 md（按版本号取最新）
2. 读取用户写在空白 `>` 位置的批注内容
3. 根据批注逐条改稿
4. **复制为新版本文件**（不覆盖旧版）：按修改幅度判断大/小版本号，文件名 `v{x}.{y}-{15字简述}.md`
5. 更新标题下版本批注块（版本号、时间 `YYYY-MM-DD HH:MM`、本版修改说明）
6. 新版本中清空已处理的批注 `>`（重新留空白 `>` 供下一轮批注）
7. 提示用户查阅新版本、继续批注

每轮循环都重复此流程，直到用户认为论文完善。

## F-4 图表生成（Claude 引导）

当 Claude 判断论文已修改得差不多（结构稳定、内容基本定稿），**主动引导用户**同意生成 Python 可生成的图和表：
```
论文内容已较稳定。其中以下图/表可由 Python 生成，
我可以建立生成脚本，你确认后我来做：
- Fig. 2（结果曲线）、Fig. 3（特征可视化）
- Table 1（主实验对比）、Table 2（消融结果）
（Fig. 1 方法框架图为手绘，需你自己绘制）
是否生成？
```

用户同意后，按以下规范生成：

### 图：`notebooks/image.ipynb`

- 新建 `notebooks/image.ipynb`
- **一个单元格生成一张图**
- 每个单元格：
  - 详细注释说明：这是 **Fig. 几**、对应**图注**是什么、**这张图如何理解**
  - 代码生成该图
  - 支持输出到 `notebooks/image/` 下，保存为 **SVG**，默认文件名 `Fig_1.svg`（按编号）
  - **输出 SVG 的代码默认被注释掉**（用户需要时取消注释运行）

```python
# Fig. 2 — {图注}
# 如何理解：{这张图展示什么、横纵轴含义、关键趋势}
# ... 绘图代码 ...
# fig.savefig('notebooks/image/Fig_2.svg', format='svg', bbox_inches='tight')  # 默认注释，取消注释以输出
```

### 表：`notebooks/table.ipynb`

- 新建 `notebooks/table.ipynb`
- **一个单元格生成一张表**，生成的表格式与论文中要求的完全一致
- 每个单元格详细标注：**表名**、**每一行每一列的含义**

```python
# Table 1 — {表名}
# 行：{每行含义}；列：{每列含义}
# ... 制表代码（输出与论文格式完全一致的表）...
```

> 手绘图（如方法框架图）不进 ipynb，由用户自行绘制，正文图位保留图注并注明"手绘"。
